# 저장소 가이드라인

## 프로젝트 구조 및 모듈 구성
이 저장소는 현재 비어 있으므로, 아직 강제되는 애플리케이션 레이아웃이 없습니다. 프로젝트가 도입되는 과정에서 최상위 디렉터리는 최소한으로 유지하고, 책임(역할)별로 코드를 그룹화하세요.

권장 구조:
- 애플리케이션/라이브러리 코드는 `src/`
- `src/`를 미러링하는 자동화 테스트는 `tests/`
- 이미지나 픽스처 같은 정적 파일은 `assets/`
- 설계 노트, 아키텍처 결정, 사용 가이드는 `docs/`

예시:
src/
tests/
assets/
docs/

## 빌드, 테스트, 개발 명령어
아직 빌드/테스트/로컬 실행 명령어가 설정되어 있지 않습니다. 도구(툴링)를 추가하면, 이곳과 주요 프로젝트 README에 이를 문서화하세요.

선호하는 패턴:
- 기본 테스트 진입점은 `make test` 또는 `npm test`
- 프로덕션 빌드는 `make build` 또는 `npm run build`
- 로컬 개발은 `make dev` 또는 `npm run dev`

저장소 루트에서 동작하는 명령어만 추가하세요.

## 코딩 스타일 및 네이밍 규칙
처음부터 일관된 포맷을 사용하세요. YAML/JSON/Markdown 리스트는 2칸 들여쓰기를 선호하고, 기술 스택이 정해진 뒤에는 소스 파일은 해당 언어의 표준 들여쓰기를 따르세요. 이름은 의미가 분명하게 짓고, 모듈은 한 가지 책임에 집중하도록 유지하세요.

네이밍 가이드:
- 디렉터리: 소문자, 필요 시 하이픈 사용
- 소스 파일: 언어 관례에 맞춤(`snake_case.py`, `kebab-case.ts` 등)
- 테스트: 소스 이름을 미러링(예: `tests/test_api.py` 또는 `api.test.ts`)

포매터와 린터를 초기에 도입하고, PR을 열기 전에 실행하세요.

## 테스트 가이드라인
아직 테스트 프레임워크가 설정되어 있지 않습니다. 첫 기능 작업부터 테스트를 함께 추가하고, 테스트 경로는 소스 경로와 정렬되게 유지하세요. 느린 통합 테스트 커버리지를 추가하기 전에 빠르고 반복 가능한 단위 테스트를 우선하세요.

최소 기대사항:
- 모든 신규 기능에는 자동화 테스트가 포함되어야 함
- 가능한 경우 모든 버그 수정에는 회귀 테스트를 포함해야 함

## 커밋 및 풀 리퀘스트(PR) 가이드라인
이 저장소에는 아직 Git 히스토리가 없어 기존 커밋 컨벤션을 추론할 수 없습니다. `Add API client`, `Create test scaffold`처럼 짧고 명령형(동사로 시작하는) 커밋 메시지를 사용하세요.

PR에는 다음을 포함해야 합니다:
- 변경 사항 요약
- (있다면) 연결된 이슈 또는 작업 레퍼런스
- 테스트 수행 근거 또는 테스트를 추가하지 않은 이유 설명
- UI 변경의 경우에만 스크린샷 첨부
